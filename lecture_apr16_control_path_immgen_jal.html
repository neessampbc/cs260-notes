§ 01

The Decode Cycle — How to Design ItReading register addresses directly from instruction bits · Wayin 00:00:00–00:02:32

"The first thing we glossed over is the decode cycle. For all the instructions, you can see that rs1 is bits 15 to 19, and rs2 is bits 20 to 24..."

The decode cycle answers: "how does the register file know which registers to read?" The answer is trivial in RISC-V — the register addresses are at fixed bit positions in every instruction format. No parsing logic needed.

00:00:05 ▶ Watch this moment Decode cycle — Shankar opens the lecture
00:00:32 ▶ Watch this moment Decode cycle — machine code field breakdown on board
RISC-V instruction fields — register addresses are at fixed bit positions in all formats. No decode logic needed.
00:01:09 ▶ Watch this moment Decode cycle cont. — rs1/rs2/rd bit positions
rd (bits 11:7) = destination register — connected to RF WriteAddr. rs1 (bits 19:15) → RF ReadAddr1. rs2 (bits 24:20) → RF ReadAddr2. The decode is "trivial on RISC" because these positions never move.
Exam question type: "Which instruction bits feed each register file port?" Answer: rs1=19:15, rs2=24:20, rd=11:7, opcode=6:0. These are fixed — the same for R-type, lw, sw, and beq.

§ 03

Immediate Generation (ImmGen)5 instruction formats → 5 different 32-bit immediates · Wayin 00:05:17–00:13:36

ImmGen is "a lot of combinational" logic that takes the 32-bit instruction word and produces a 32-bit sign-extended immediate. The bit-scrambling varies by instruction format — each format encodes the immediate differently to keep the register fields at fixed positions.

"Why are immediates broken up in unusual ways? It simplifies hardware implementation."
00:05:20 ▶ Watch this moment ImmGen — I-type instruction (addi/lw) immediate formation
00:05:59 ▶ Watch this moment ImmGen — S-type (sw) immediate formation
00:06:32 ▶ Watch this moment ImmGen — SB-type (beq) immediate formation
I-typeaddi, lw

12-bit signed immediate, sign-extended to 32 bits. Bits are contiguous in the instruction — easiest case.

imm[11:0] = Instr[31:20]
imm[31:12] = SignExt(Instr[31]) // sign-extend bit 31
S-typesw

12-bit immediate split across two fields — upper 7 bits and lower 5 bits — to keep rs2 at bits 24:20.

imm[11:5] = Instr[31:25]
imm[4:0] = Instr[11:7]
// Concat then sign-extend
SB-typebeq

13-bit offset (bit 0 = 0, halfword aligned). Bits scrambled to keep register fields stable.

SignExt(Concat(
Instr[31], // imm[12]
Instr[7], // imm[11]
Instr[30:25], // imm[10:5]
Instr[11:8], // imm[4:1]
0) // imm[0] = 0
UJ-typejal

21-bit PC-relative offset (halfword aligned, bit 0 = 0). Most scrambled format — keeps register positions stable.

SignExt(Concat(
Instr[31], // imm[20]
Instr[19:12], // imm[19:12]
Instr[20], // imm[11]
Instr[30:21], // imm[10:1]
0) // imm[0] = 0
00:06:54 ▶ Watch this moment ImmGen — UJ-type (jal) immediate formation
The ImmGen output feeds the ALU's second input (when ALUSrc=1) AND the branch/jump adder. For JAL and beq, the immediate goes directly to the adder computing the target PC. Know the bit positions by instruction format — the scrambling is intentional to keep rs1/rs2/rd stable.
§ 04

ImmGen Hardware ImplementationMinimizing gates with 2-to-1 MUXes · Wayin 00:10:42–00:16:38

With 5 instruction types and 32 output bits, the naive ImmGen would be huge. The key insight: most bits of the 32-bit immediate come from at most 2 sources depending on instruction type. So each bit can be implemented with a small 2-to-1 MUX (or 3-to-1/4-to-1 for the exceptions).

"We like smaller muxes. Two-to-one muxes. The goal is to minimize hardware by using smaller muxes, specifically two-to-one muxes, for all 32 bits of the immediate generator."
00:10:55 ▶ Watch this moment ImmGen hardware minimization — MUX approach introduced
00:13:46 ▶ Watch this moment MUX minimization — 2-to-1 MUX for each bit of the immediate
MUX internals from lecture notes p.3 — F = c·Y + c'·X for 2-to-1. ImmGen preference: 2-to-1 MUXes throughout, minimizing AND gates.
00:13:50 ▶ Watch this moment MUX minimization — final hardware diagram
Example: Computing imm[10] for Each Instruction Type

From notes page 2: bit 10 of the immediate is a good example of how each bit is a MUX output:

Each of the 32 imm bits follows this pattern: a small MUX where the select comes from the opcode, choosing between instruction fields or a constant 0.
The ImmGen is entirely combinational. The opcode (bits 6:0) controls which instruction format to decode, selecting the right bit sources for each of the 32 immediate output bits. Each output bit is a small MUX — mostly 2-to-1.
§ 05

Adding JAL to the Single-Cycle DatapathJump and Link — two operations, two datapath modifications · Wayin 00:19:23–00:26:13

"jal — jump and link: store PC+4 in rd (of RF). Link part: store function of label in PC. Now we will modify the circuit."

jal rd, Label does two things simultaneously: writes PC+4 (return address) to register rd, and jumps to PC + UJ-imm. The existing Figure 4.21 datapath handles neither of these — two additions are required.

▶ Watch: JAL datapath modification — Shankar starts — 19:23
JAL format from notes p.3: |Label (31→12)| rd (11:7) | opcode (6:0)|. ImmGen handles the UJ scrambling — no new hardware needed for the immediate itself.
Two Required Datapath Modifications
🔧
Modification 1 — MemToReg MUX: 1-bit → 2-bit (add PC+4 input)
The existing MemToReg MUX selects between ALU output (0) and DataMem output (1) for the RF write data. JAL needs a third option: PC+4 (the return address). Expand to a 3-input MUX with a 2-bit select. New values: 00=ALU result, 01=DataMem, 10=PC+4. From notes p.3: readdata → (2 M U X) — the readdata MUX gets the new PC+4 input.
🔧
Modification 2 — Branch MUX: 1-bit → 2-bit (add unconditional jump)
The existing branch MUX: (Branch AND Zero)=1 → take branch target, else → PC+4. JAL is unconditional — no condition needed. Expand the MUX to 2-bit select. New case: 10 = always take the JAL target address (PC + UJ-imm from ImmGen). The ImmGen already produces the correct UJ-type immediate — no extra hardware for address computation.
JAL requires two MUX expansions: (1) Branch/PC MUX gains a 3rd input for the unconditional JAL target. (2) MemToReg MUX gains PC+4 as a new input for writing the return address to rd.

▶ Watch: Branch MUX modification for JAL — 22:13 ▶ Watch: Existing MUX modifications — 1-bit to 2-bit — 24:33
JAL is the first instruction where ALUSrc and ALUOp are true don't-cares (the ALU is simply not used). The two MUX expansions are the key hardware changes — both become 2-bit selects. The ImmGen already supports UJ-type and requires no changes.

Adding "push reg" to the DatapathA new RISC-V instruction — sp is register 2 · Wayin 00:27:20+

"Let's say we want to push reg. sp is reg 2. Subtract 4 from RF[2], store back into RF[2]. Store reg (from RF) into MEM[new sp]."

RISC-V has no native push instruction (unlike x86). Shankar designs one from scratch, walking through the English description → machine code format → datapath modifications → control signals.

📖 Figure 4.21 Annotated — Push Reg Modifications (class notes + WhatsApp GC)

This is the annotated Figure 4.21 from class showing all push reg modifications. Key annotations: "we want to (get sp addr) read reg 1" on the left, "RF[2] (sp addr) store reg @ new sp" on the store path, "for −4" on the new ALUSrc input, and the expanded control table showing Push row at bottom (ALUSrc=10, ALUOp=SUB).

Machine Code Format
Push reg machine code: rs2=register to push [24:20], hardwired 4 at [24:20] position for ALU input, rs1=2 (sp) at [19:15], rd=2 (sp) at [11:7] for writeback.
Three Design Steps
  • Step 1
    Subtract 4 from rs2 [15:19] — get new sp address. Allows getting both data and address into memory. From fig: rd1, rd2 → ALU → address → Mem write. The constant 4 is hardwired into a new input of the ALUSrc MUX.
  • Step 2
    Store into RF[2] (sp) — input 2 (constant sp address) into the write register MUX. This updates sp with the decremented value.
  • Step 3
    Finish the control path — add a row to the control table, update any MUX control bits that changed width.
Push reg: ALUSrc MUX gains a 3rd input (constant 4 for sp decrement). ALUSrc becomes 2-bit. New value 10 = "subtract 4 from sp." All existing instructions update: 0→00, 1→01.

Updated Control Signal TableFigure 4.22 adapted + Push row added · singlecontrol.pdf

The annotated WhatsApp image shows the control table with the Push row filled in. Every existing instruction's ALUSrc values must be updated now that the signal is 2-bit (0→00, 1→01). From the annotated diagram: Push ALUSrc=10, MemToReg=0, RegWrite=0, MemRead=0, MemWrite=1, Branch=0, ALUOp=SUB.

R-type000 1000OP
lw011 1100ADD
sw01X 0010ADD
beq00X 0001SUB
jalXX10 1000XX
push100 0010SUB
ALUSrc is now 2-bit. Original values updated: old 0 → new 00 (register B), old 1 → new 01 (sign-extended immediate). Push uses new value 10 = hardwired constant 4. All columns that had 1-bit selects must be updated in the control table when you expand a MUX.
Nava Nizard: "How does the push instruction affect the control signals for other instructions?"
Alex V-H: "Add a row for push, add a column for every mux or control you add. If you edit any mux, you need to update the table. Like in class he added an input, so he had to add a bit to the control wire — 0→00, 1→01, etc."
When you add an input to a MUX and make its select wire wider, you MUST update every row in the control table for that signal. Old single-bit values become the low-order bit of the new multi-bit value (0→00, 1→01). New rows get new values for the new inputs.

Single-Cycle Datapath Modification — Add push reg (HW3 / B3)

What the homework asks: Modify Figure 4.21 (singleData.pdf) and Figure 4.22 (singlecontrol.pdf) to support a new push reg instruction. The homework version specifies bits 7–11 contain the value 4 (rather than Shankar's in-class version where 4 was hardwired into the MUX directly).

How it connects to this lecture: This homework is a direct application of everything in §05–§07 above. The key skill is Shankar's design methodology: write the English description first, then derive the machine code format, then trace what new paths are needed in the datapath, then update the control table. The JAL exercise in §05 is the template — push reg follows the same three-step pattern.

The class vs HW difference (from groupchat): In class, Shankar hardwired the constant 4 directly into the new ALUSrc MUX input. In the homework, bits 11:7 of the instruction contain 4 — so instead of a hardwired wire, you connect the IR[11:7] wire to that MUX input. The control table update is the same in both cases: ALUSrc becomes 2-bit.
