§ 01

Push Instruction — SetupWhat push does · register encoding · 00:00:09 – 00:02:48

"One of the things we have to do in push is we have to subtract four from the stack pointer. We look up our green card. We realize that the register is the stack pointer." 00:00:26
"So we want register two. We want to subtract four from register two." 00:00:48
02:51 ▶ Watch this moment Board: “Ex push reg” register diagram + step ③ ALUOUT ← sp−4

A push rd instruction does two things: (1) stores the value in rd at memory address pointed to by the new decremented stack pointer, and (2) updates the stack pointer: sp ← sp − 4.

In RISC-V, sp = register 2. After Decode (cycle ②), register A already holds sp (because IR[19:15] encodes x2 for push). Register B holds the data to push (IR[24:20]). This means cycle ③ can compute sp−4 immediately — no reload needed.

"After decoding, the stack pointer will be in input A, allowing for the computation of sp minus four in cycle three." 00:01:32
Instruction encoding: bits IR[19:15] = sp (x2) = rs1 field. Bits IR[11:7] = rd field = the destination register (where the new sp gets written back). Bits IR[24:20] = the register being pushed.
§ 02

Deliberate MistakeShankar writes RF[2] ← sp−4 in cycle 3 · 07:26

At timestamp 7:26, Shankar wrote RF[2] ← sp−4 for cycle ③. A student caught it. Your handwritten note flags it: “#this would be wrong (more in video)”.
What Shankar wrote first (cycle 3)
③
RF[2] ← sp − 4
↤ WRONG — student caught this

Why it’s wrong: Writing directly to RF in the same cycle as the ALU computes the value creates a long combinational path: ALU output → RF write port, without going through the ALUOUT inter-stage register. This violates the core multicycle principle.

"The goal is to ensure anything from the ALU goes to ALUOUT and then to the register file in the next cycle, to avoid large path lengths and maintain clock speed." 00:04:02
Any value computed by the ALU must be stored in ALUOUT first. It can only be written to the register file in the next cycle. If you write RF in the same cycle as the ALU op, it’s always wrong in multicycle RTL.

Wrong-way 7-cycle design (timestamps 7:30–7:32)
①
// Fetch
②
// Decode — A=sp, B=push-register (already here!)
③
ALUOUT ← A  // “Assume ALU has a pass-A instr.” ← needs new opcode = datapath change!

⑦

Three problems: (1) Cycle ③ assumes a pass-A ALU opcode that doesn’t exist → requires a datapath change. (2) Cycle ④ reloads sp even though A already held it from Decode. (3) Cycle ⑥ reloads push-register even though B held it from Decode. Net: 7 cycles for what needs only 4.
"Your first try will be a little inefficient, because you’re doing the different parts of the instructions separately, and then you’ll figure out that you can combine them." 00:09:25
§ 04

Correct Push RTL — 4 CyclesReuse A and B from Decode · 00:04:02 – 00:09:54

Push — correct 4-cycle RTL
①
Fetch
IR ← InstrRAM[PC]  ;  PC ← PC+4
②
Decode
A ← RF[IR[19:15]]  ;  B ← RF[IR[24:20]]
// A = sp (x2), B = push-register — both FREE from existing decode logic
③
ALU Compute
ALUOUT ← A − 4  ;  sp ← A − 4
// “//” = simultaneous. ALUOUT captures new sp; sp register updates. Notes: “ALUOUT // sp ← sp−4”
ALUSrcA=1 ALUSrcB=01 (const 4) ALUOp=SUB
④
Writeback + Store
RF[IR[11:7]] ← ALUOUT  ;  MEM[ALUOUT] ← B
// Both happen simultaneously. Notes: “RF[IR[1:7]] ← ALUOUT” (IR[11:7] in full)
RegWrite=1 MemtoReg=0 MemWrite=1 IorD=1
Cycle ④ has two simultaneous statements. RTL is parallel: all right-hand sides read values from the start of the cycle. So RF write and MEM write both fire at once. This is legal and intentional.
"Make bits 11 through 7 also two bits to avoid datapath changes and ensure IR[11:7] gets ALUOUT." 00:05:29
"So now you have the address in the ALUOUT, the data is in B." 00:08:48
📖 Figure e4.5.4 — Complete Multicycle Datapath (multiData.pdf)

Push uses this datapath as-is — no new wires. Cycle ③ path: A register → MUX(ALUSrcA=1) → ALU(SUB) → ALUOUT register. Cycle ④ dual-write: (a) ALUOUT → MUX(MemtoReg=0) → RF write port; (b) B register → Memory write (IorD=1, MemWrite=1). The IorD MUX selects ALUOUT as the memory address. Hatched boxes = clocked sequential elements (A, B, ALUOUT, MDR, IR, PC).

New FSM States for PushAdditions to Figure e4.5.12 · 00:10:02 – 00:13:54

14:18 ▶ Watch this moment Projector: partial FSM state diagram — MemtoReg=0 state visible
Push shares States 0 (Fetch) and 1 (Decode/RegFetch) with all instructions. After State 1, an Op = push arc routes to two new states.
State 0 (Fetch) → State 1 (Decode) — same as all instructions State 1 → [Op = push] → State P3 (new)   Signals: ALUSrcA=1, ALUSrcB=01 (const 4), ALUOp=SUB State P3 → (unconditional) → State P4 (new)   Signals: MemtoReg=0, RegWrite=1, IorD=1, MemWrite=1 State P4 → State 0
Handwritten notes: “ALUSrcB = GT” is the notation for selecting constant 4 on the B MUX (= ALUSrcB=01 in Fig e4.5.12 terms). The state diagram in your notes shows these two new states explicitly with “added to figure” annotation.
📖 Figure e4.5.12 — Multicycle FSM (multiControl.pdf)

Base FSM before push addition. Push adds two new nodes branching off State 1 (Op=‘push’ arc). Structural analogs: State 5 (sw memory write) and State 7 (R-type writeback). The unique aspect of push is combining RegWrite=1 and MemWrite=1 in a single state (P4), which is distinct from all existing states.

HW4 asked you to add jalr and stw to this same FSM. Push follows the identical pattern: (1) identify which state the new opcode branches from, (2) design the new state chain, (3) assert minimum control signals per state, (4) connect back to State 0.

§ 10

State Machine Design Review00:26:49 – 00:33:26

Shankar asked how many students remembered state machines from a prior course. Quick review of how any FSM maps to hardware — the same theory underlying the multicycle control unit.

Any finite state machine maps to hardware with three components:

1. State register — clocked D flip-flops. Width = ⌈log₂(num states)⌉ bits. For 9-state multicycle FSM: 4 bits. For 4-state example: 2 bits [s1, s0].

2. Next-state logic — combinational circuit. Inputs: current state + opcode. Output: next state value. Minimize with Quine-McCluskey or K-maps.

3. Output logic — combinational circuit (Moore machine). Input: current state. Output: the control signals (MemRead, RegWrite, IorD, etc.).

      Opcode / conditions            |            v     +—————————+     | Next-state |<————Q (current state)     | logic (comb.) |     +—————————+            | D (next state)            v     +—————————+     | State register | ← Clock edge     | (D flip-flops) |     +—————————+            | Q (current state)            v     +—————————+     | Output logic |————> Control signals     | (comb.) |     (MemRead, IorD, RegWrite ...)     +—————————+
Connection to Appendix C Fig C.5.1: Each row of the microcode field-to-control-signal mapping table = one output function g() in the output logic block. The full multicycle control unit implements all these functions using this exact FSM hardware structure.
If asked to implement an FSM in hardware: (1) determine state bit count, (2) write next-state truth table, (3) write output function truth table per signal, (4) minimize with K-map. Same process as building the multicycle control unit.
