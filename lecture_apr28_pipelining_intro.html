§ 01

Pipelining Goal

What pipelining is — and what it is not

Pipelining's goal is to increase throughput. Not to make any single instruction execute faster — that's latency, and pipelining doesn't improve it. Pipelining works by overlapping the execution of many instructions so that the processor is doing useful work in every cycle.

Shankar uses a laundry analogy in class (the same one in your textbook §4.6): if you have 6 loads of laundry and each load takes 2 hours, doing them one at a time takes 12 hours. But if you overlap wash/dry/fold stages, you can finish all 6 in much less time — even though each individual load still takes the same 2 hours.

LATENCY
seconds / instruction
Time to execute a single instruction from start to finish.
Laundry: 2 hrs for one load of laundry
THROUGHPUT
IPS  (instructions per second)
Number of instructions completed per unit time.
Laundry: 5 hrs for 6 loads (overlapping)
KEY POINT
Pipelining does not reduce latency of any single instruction — it actually adds a tiny bit of overhead from pipeline registers. What it does is let the processor start a new instruction every cycle, which dramatically increases throughput.

§ 02

Latency vs. Throughput

The two metrics — never confuse them

These two terms appear constantly in this course. Shankar is strict about the distinction: any time you see "pipelining improves performance," the correct interpretation is that it improves throughput, measured in IPS (instructions per second) or IPC (instructions per cycle). Latency per instruction is a fixed cost of the instruction's complexity, not something pipelining touches.

§ 03

The 5-Stage RISC-V Pipeline

The five stages every RISC-V instruction passes through
IF
Instruction Fetch
ID
Decode / Reg Read
EX
Execute / ALU
MEM
Memory Access
WB
Write Back

IF — Read the instruction from instruction memory using the PC. PC advances to PC+4.

ID — Decode the opcode, read the register file for rs1 and rs2, sign-extend the immediate. This is where ImmGen runs (Apr 16).

EX — The ALU performs the operation. For branches, the target address is computed here. For R-type, this is the final result. For loads/stores, this computes the memory address.

MEM — Data memory is read (lw) or written (sw). For instructions that don't touch memory, this stage is effectively a pass-through.

WB — Write the result back to the register file. For lw this is the value read from memory; for R-type it's the ALU result.

WHY 5?
These 5 stages map directly onto the critical path of the single-cycle RISC-V datapath you built in Apr 14–21. The longest instruction (lw) exercises all 5 stages. Pipelining lets every stage work on a different instruction simultaneously.

§ 04

Issue 1A — Equal-Time Stages

The first design challenge: making stages the same duration

For a pipeline to work cleanly, every stage must take the same amount of time. The clock cycle is set by the slowest stage. If MEM takes 100ps and everything else takes 50ps, the whole processor runs at the 100ps clock — even on cycles where MEM does nothing.

This is Issue 1A from your notes: how do you break an instruction into equal-time stages? Shankar notes this is also apparent in multicycle designs (which you saw in Apr 21), where unequal stage times force you to set the clock period conservatively.

RISC was designed partly to make this easier — simpler, more regular instructions create more balanced pipeline stages. CISC instructions (like x86) have wildly varying complexities, making equal-time staging much harder.

EXAM TIP
If asked "what determines clock speed in a pipelined processor?" — the answer is the latency of the slowest pipeline stage, not the average. This is the same critical-path logic from Apr 21, applied per-stage.
§ 05

Hazard Types

Issues 2A, 2B, 2C — structural, data, and control hazards

Hazards are situations where the next instruction can't execute in the next clock cycle as expected. Shankar taxonomizes them as Issues 2A, 2B, and 2C in your notes. All three types require solutions or they silently produce wrong results.

ISSUE 2A

Structural Hazard

Hardware can't support a particular combination of instructions at the same time.

Example: two instructions both need to use the adder in the same clock cycle. The hardware has one adder — it can't be in two places at once.

In the infinite-stage pipeline example from your notes, this constraint disappears (infinite resources assumed). In real designs it's a physical resource conflict.

FROM NOTES — Infinite pipeline example
L2: sub x2, x1, x3  ·  L2: and x12, x2, x5  ·  L3: or x13, x6, x2  ·  L4: add x14, x2, x2  ·  L5: sw x15, 100(x2)  ·  L6: lw x7, 200(x14)
These share registers — assume infinite pipeline stages for this analysis.
ISSUE 2B

Data Hazard

Results of an instruction are needed by a later instruction before they're available.

The classic example from your notes:

add s1, s2, s3 # writes s1 in WB (cycle 5) sub t0, s1, t1 # reads s1 in ID (cycle 3) — too early!

The sub needs s1 during its ID stage (cycle 3), but add doesn't write s1 until its WB stage (cycle 5). Without intervention, sub reads a stale value.

A load-use hazard is the nastiest case: a lw followed immediately by an instruction using the loaded value. Even forwarding can't fix this — you always need at least 1 stall cycle.

ISSUE 2C

Control Hazard

Results of an instruction determine which instruction to fetch next.

The example from your notes:

beq t0, t1, L2 # do we jump to L2, or go to L1? L1: ... # already fetched — might be wrong! L2: ... # or should we be here?

By the time the processor knows whether the branch is taken, it has already fetched (and possibly started executing) the next instruction. If the branch is taken, those instructions in the pipeline are wrong and must be flushed.

§ 06

Solutions to Hazards

Shankar's taxonomy from your notes — four approaches, three categories
Issue(s) Approach Who handles it Notes
1A Design ISA to match hardware ISA Design Easier with RISC — simple, regular instructions create balanced pipeline stages. RISC-V was designed with pipelining in mind.
2A 2B 2C Design ISA to minimize hazards ISA Design e.g., RISC-V has no implicit register dependencies in its encoding; operands are always in fixed bit positions.
2B 2C Make compiler not generate hazardous sequences Compiler The compiler can reorder instructions to put independent work between dependent instructions, eliminating stalls without hardware changes.
2A 2B 2C Have HW handle hazards (stalls + forwarding + branch prediction) Hardware The most general solution. Forwarding routes ALU results directly to the next instruction's input without waiting for WB. Stalling inserts bubble cycles. Branch prediction guesses the control flow.
SHANKAR'S DESIGN PHILOSOPHY
In class, Shankar emphasizes that RISC architectures were intentionally designed to make pipelining easier. The regular encoding, load-store architecture, and fixed instruction size are all choices that reduce hazard frequency before hardware ever gets involved.

0.4 IPC << 1.0 IPC (ideal, no stalls) — stalls are expensive
WHAT THIS TELLS US
The ideal pipelined throughput is 1 IPC — one instruction completed per cycle. Every stall cycle eats into that. Here, 9 stall cycles for 6 instructions cuts throughput by more than half. Real processors use forwarding and branch prediction to push toward 1 IPC.
§ 08

Practice Problems

Five problems — click Reveal Answer after you try each one
PRACTICE Based on class material · §4.6
P1 MULTIPLE CHOICE What does pipelining improve?

Which of the following best describes the effect of pipelining on processor performance?

A Latency of a single instruction
B Throughput — number of instructions completed per unit time
C Both latency and throughput equally
D Neither — it only simplifies the control unit
ANSWER — B

Throughput only. Pipelining does not reduce the time it takes any single instruction to complete — latency stays the same or increases slightly due to pipeline register overhead. What pipelining does is overlap multiple instructions so that one completes every cycle (ideally), dramatically increasing throughput. Shankar's laundry analogy: each load of laundry still takes 2 hrs, but you can do 6 loads in far less than 12 hrs by overlapping stages.

P2 IDENTIFY HAZARDS Classify each hazard in this sequence

For each adjacent pair of instructions below, state whether there is a hazard, and if so what type (structural, data, or control). Assume a standard 5-stage RISC-V pipeline.

L0: add s1, s2, s3 L1: sub t0, s1, t2 L2: lw s1, 0(t0) L3: beq t0, s1, L5
ANSWER

L0 → L1: Data hazard. L0 writes s1 (available after WB, cycle 5). L1 reads s1 in ID (cycle 3). Without forwarding, L1 reads a stale value.

L1 → L2: Data hazard. L1 writes t0; L2 uses t0 as a base address. Same problem — write-before-read timing conflict.

L2 → L3: Load-use data hazard (the worst kind). L2 is a lw — its result (s1) doesn't exist until the end of MEM. L3 needs s1 in ID. Even with forwarding, you still need 1 mandatory stall cycle here. There is also a control hazard from the beq — by the time we know whether it branches, instructions after it may already be in the pipeline.

P3 TRUE / FALSE Stalls and the ideal pipeline

True or False: In a pipelined processor with no hazards (no stalls, no branches), the throughput converges to exactly 1 IPC once the pipeline is full.

Explain your reasoning.

ANSWER — TRUE

True. Once the pipeline is full (after the first 4 cycles filling up all stages), one instruction completes every clock cycle — exactly 1 IPC. The 5-cycle "fill" cost at the beginning amortizes to nothing over a long program. Every stall cycle (from any hazard) adds one wasted cycle and reduces IPC below 1. The gap between actual IPC and 1.0 IPC is a direct measure of how much damage hazards are doing. In class, Shankar's 6-instruction example produced 0.4 IPC — 60% of cycles were wasted on stalls.

P4 CALCULATION Throughput with stalls

A program with 10 instructions executes on a 5-stage RISC-V pipeline. Due to data hazards, 4 stall cycles are inserted. Due to a branch misprediction, 2 more cycles are wasted. The clock runs at 2 GHz.

a) How many total cycles does the program take?
b) What is the throughput in IPC?
c) What is the total execution time in nanoseconds?

ANSWER

a) Total cycles: Without stalls, a 10-instruction program takes 10 + 4 = 14 cycles (10 instructions + 4 cycles to fill the pipeline). With 4 data stalls and 2 branch penalty cycles: 14 + 4 + 2 = 20 cycles.

b) IPC: 10 instructions / 20 cycles = 0.5 IPC. (Better than Shankar's 0.4 IPC class example, but still far below the ideal 1.0 IPC.)

c) Execution time: Clock period = 1 / 2 GHz = 0.5 ns. Total time = 20 × 0.5 ns = 10 ns.

P5 SHORT ANSWER Which solution fits which hazard?

Match each scenario to the most appropriate solution from Shankar's taxonomy (ISA design, compiler, hardware):

1. A compiler reorders instructions to place independent work between a lw and the instruction that uses its result.
2. RISC-V always puts rs1 in bits [19:15] and rs2 in bits [24:20], so hardware can detect dependencies early in the pipeline.
3. The pipeline detects that a lw is followed immediately by an instruction using the loaded value, and inserts a stall bubble automatically.
4. A branch predictor guesses the next PC so the pipeline doesn't have to wait for the branch outcome.

ANSWER

1. Compiler — instruction reordering to eliminate load-use stalls without hardware changes. Solves 2B (and sometimes 2C).

2. ISA Design — fixed register field positions make it cheap for hardware to detect hazards. This is part of why RISC was designed the way it was. Solves 2ABC.

3. Hardware (stalling) — the hazard detection unit inserts a bubble. Solves 2B/2C.

4. Hardware (branch prediction) — speculative execution to handle control hazards. Solves 2C.
