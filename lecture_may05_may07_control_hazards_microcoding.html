§ 01

Control Hazards

The branch problem — when we don't know what to fetch next

A control hazard occurs whenever the processor fetches an instruction before it knows whether a branch was taken. By the time the beq (or any branch) resolves in the EX stage, two instructions have already been fetched behind it — and if the branch is taken, those fetched instructions are wrong.

This is Issue 2C from Apr 28. The three classic solutions are:

SOLUTION 1 — STALL (always safe, always slow)
After fetching a branch, freeze the pipeline until the branch outcome is known (end of EX stage). Insert bubble cycles for every branch. Cost: 1–2 wasted cycles per branch. With 15% branch frequency this is a significant throughput hit.
SOLUTION 2 — PREDICT NOT TAKEN
Always assume the branch is not taken and keep fetching sequentially. If the prediction is right: no penalty. If wrong: flush the wrongly-fetched instructions (insert bubbles). Simple hardware, works well when branches are rarely taken.
SOLUTION 3 — BRANCH PREDICTION (dynamic)
Hardware tracks branch history and predicts taken/not-taken based on past behavior. Modern CPUs achieve 95%+ accuracy. A misprediction typically costs 10–20 cycles. This is what the HW4 Newer Material branch prediction problem tests.

§ 02

The Problem: Implementing the FSM in Hardware

The multicycle controller is a state machine — how do you actually build it?

The multicycle RISC-V controller from Apr 23 is a finite state machine. When it comes time to build it in silicon, you have two options. Shankar walks through both and explains why one is practically unusable.

APPROACH 1 — BAD

Directly encode as a sequential circuit

  • Too many states
  • Hardware too complicated (esp. in VLSI)
  • Difficult to debug and modify
  • Difficult to layout in VLSI/silicon
APPROACH 2 — GOOD

Encode as a micro-program

  • Microcode generates control signals
  • Hardware just executes the micro-program
  • Easier to debug and modify
  • Scales cleanly to VLSI layout
SHANKAR'S FRAMING
Instead of baking every control decision into combinational logic (which gets enormous fast), you store the control signals as a program in a small ROM. The hardware reads and executes that program one microinstruction at a time — just like a CPU executes instructions, but one level down.

§ 03

Approach 2: The Micro-program

Encoding control logic as a small program executed by dedicated hardware

In the micro-program approach, each state of the multicycle FSM becomes a microinstruction stored in a small control memory (microcode ROM). Each microinstruction contains the values of all control signals needed in that state — ALU operation, register reads/writes, memory access, next-state logic.

The hardware that runs this micro-program is simple: a micro-PC (µPC) that addresses the control memory, plus address-select logic that chooses the next microinstruction.

Microcode generates the signals sent to the datapath. The datapath itself does not change — only the source of its control signals changes (from hardwired combinational logic to a ROM read).

§ 04

The micro-PC and Address Select Logic

How the micro-program knows where to go next

The micro-PC (µPC) works exactly like the regular PC, but for microinstructions. It holds the address of the current microinstruction in the control ROM. After each microinstruction executes, the address-select logic chooses the next µPC value from three options:

OPTION 0 — FETCH (µPC = 0)
Reset to address 0. This is the Fetch microinstruction — the start of every instruction's execution. Fires after the last microinstruction of any instruction completes.
OPTION 1 — µPC + 1 (sequential)
Move to the next microinstruction in sequence. Used for the steps within an instruction (e.g., Fetch → Decode → Execute).
OPTION 2 — DISPATCH TABLE (lookup)
After Decode, the opcode determines which instruction-specific microcode to jump to. A dispatch table (look-up table) maps opcodes to µPC addresses. There are 2 "branch" states in the FSM that use this — these become dispatch table lookups in microcode.
KEY RULE FROM NOTES
µPC must reset to 0 after every instruction. Unlike the regular PC (which increments continuously), µPC is local to the current instruction's execution and always restarts at Fetch when done.

§ 05

Microinstruction Fields

What each microinstruction contains — the three field groups

Each row in the microcode ROM is a microinstruction. It has multiple fields — each field controls a different part of the datapath for that cycle. From your notes, Shankar identifies three main field groups:

FIELD 1 — ALU CONTROL
What operation the ALU performs this cycle. Same values as the ALU control signal in single/multicycle: add, sub, and, or, slt, etc.
FIELD 2 — SRC1 + SRC2
What the two ALU inputs are. SRC1 can be the register file output A, the PC, etc. SRC2 can be register file output B, the immediate, the constant 4, etc.
FIELD 3 — REGISTER CONTROL
Read 2 register file entries into regA and regB. Write ALU result to reg file. Write contents of ALUOut to the destination register. Controls all register file reads and writes for this microinstruction.
§ 06

Microcode Sequences

The actual microcode for each instruction group — from your notes (App C.4–C.5)

Each macroinstruction (lw, add, beq, etc.) expands into a sequence of microinstructions. The first two are always shared (Fetch and Decode), then the Dispatch jumps to the instruction-specific microcode. From your notes (cross-referenced with App C pages 88+):

// Shared prefix — all instructions start here Fetch: Add & PC & 4 & Read & ReadPC & ALU & Seq // ALU computes PC+4, reads instruction memory, µPC → µPC+1 Decode: Add & PC & Imm & Read & _ & _ & Dispatch1 // Read reg file, compute branch target (PC+imm) // "Don't care" conditions for most fields // Dispatch1: jump to instruction-specific microcode via opcode lookup // R-type instructions (add, sub, and, or, etc.) R-Exec: Func & A & B & _ & _ & Seq & _ & WriteALU & _ & _ & Fetch // Func = ALU op from funct3/funct7 field // A and B from register file (regA, regB) // WriteALU: write ALUOut to rd // Fetch: µPC resets to 0 (done, start next instruction) // Memory instructions Mem: ... (address compute → memory access → writeback steps) // Branch instructions Br: ... (compare rs1, rs2 → update PC if taken)
NOTE FROM SHANKAR
The textbook format (App C) uses MIPS, not RISC-V. Shankar adapted it to RISC-V in class. The field names and sequence shown in your notes reflect the RISC-V version — don't copy directly from the textbook tables without adjusting for RISC-V differences.
HOW TO READ A MICROINSTRUCTION ROW
Each &-separated value is one field. _ or blank = don't care (irrelevant for this step). Seq = µPC+1 (sequential). Fetch = µPC reset to 0. Dispatch1 = jump via lookup table. WriteALU = write ALU result to register file.
§ 07

Microcode-Related Issues & RISC vs. CISC

Why complex instructions exist, and why RISC argues against them

One major advantage of microcode is that you can implement a complex instruction entirely in microcode without adding hardware. Classic CISC examples from your notes:

// CISC instructions implemented in microcode MOVS // x86 — "move string": copies a block of memory in one instruction // expands to many microinstructions: loop, copy, decrement, branch SOBGTR // VAX / LSI-11 — "subtract one and branch if greater" // combines subtraction + comparison + branch into one macroinstruction
THE RISC REBUTTAL (from notes)
These complex instructions can also be implemented using assembly language (or a compiler). If you can write a loop in assembly that does what MOVS does, you don't need a special instruction for it. RISC's argument: let the compiler handle complexity; keep the ISA simple. This is the foundation of the RISC vs. CISC debate that the schedule lists as a separate topic.
Related topic RISC vs. CISC — schedule (multiple dates) Example: x86 = CISC, RISC-V = RISC
§ 08

Pros & Cons of Microcode

Why it was adopted, why RISC moved away from it
PROS
  • No fetch-decode overhead for complex operations → increased code density (from cache perspective)
  • Can implement any complex instruction without new hardware
  • Easy to fix bugs post-manufacture (update ROM, not silicon)
  • Cleaner separation: datapath hardware is fixed; control changes via ROM update
CONS
  • Must support more general form of instruction → less efficient (compilers optimize simple ops better)
  • Most compilers can optimize better for most cases than hand-crafted microcode
  • How to pick useful instructions? Hard design problem
  • Upward compatibility (backward compat) — older programs must still run on new hardware, locking in old ISA decisions. Example: BCD (Binary Coded Decimal) — 4 bits per decimal digit — was kept in x86 for decades for this reason
UPWARD / BACKWARD COMPATIBILITY — KEY CONCEPT
Once you publish an ISA and software is written for it, you can never remove an instruction. x86 still supports 8086 instructions from 1978. This is why RISC-V was designed from scratch as a clean slate — no legacy baggage. The BCD example from your notes: VAX and early x86 supported BCD arithmetic natively; it's essentially unused now but removing it would break old programs.

P1 MULTIPLE CHOICE Branch prediction accuracy and CPI

A RISC-V pipeline runs at 3 GHz. 20% of instructions are branches. A branch misprediction costs 3 stall cycles. The predictor has 90% accuracy. All data hazards are handled by forwarding (no data stalls). What is the CPI?

A 1.00
B 1.03
C 1.06
D 1.20
ANSWER — C

CPI = 1 + (branch_fraction × misprediction_rate × stall_penalty)

= 1 + (0.20 × 0.10 × 3) = 1 + 0.06 = 1.06

Throughput = 3 GHz / 1.06 ≈ 2.83 BIPS. Note: the 10% misprediction rate (1 - 90% accuracy) is the key term. Higher accuracy directly reduces the penalty.

P2 TRUE / FALSE µPC vs. PC

True or False: The micro-PC (µPC) increments continuously across instruction boundaries, just like the regular PC increments across instructions.

Explain why or why not.

ANSWER — FALSE

False. The µPC resets to 0 after every instruction. The regular PC increments continuously (PC+4, or branch target) across the entire program. The µPC is local to the current instruction's microcode execution — when the last microinstruction of an instruction fires (the one that says "Fetch"), the µPC jumps back to 0 to start the shared Fetch microinstruction for the next macroinstruction. It does NOT carry over between instructions.

P3 FILL IN Next µPC value

For each situation, state what value the address-select logic gives to µPC next (choose from: 0, µPC+1, or dispatch table lookup):

a) The processor just finished the Fetch microinstruction.
b) The processor just finished the Decode microinstruction.
c) The processor just finished the last microinstruction of an R-type instruction (WriteALU).
d) The processor is in the middle of a multi-step memory instruction (e.g., address compute step).

ANSWER

a) µPC+1 — Fetch is always followed sequentially by Decode.

b) Dispatch table lookup — Decode ends with Dispatch1. The opcode is looked up in a table to find which instruction-specific microcode to jump to.

c) 0 (reset to Fetch) — The last microinstruction of any instruction returns µPC to 0 to begin the next macroinstruction's Fetch.

d) µPC+1 — Within a multi-step instruction, steps run sequentially (address compute → memory read → writeback), each incrementing µPC by 1.

P4 MULTIPLE CHOICE Why dispatch tables at Decode?

Why does the Decode microinstruction use a dispatch table (look-up table) to choose the next µPC, rather than µPC+1?

A Because Decode always takes more than one cycle
B Because different instruction types need different microcode sequences after Decode, and the opcode determines which sequence to run
C Because Decode modifies the µPC register directly
D Because the register file read takes too long to fit in one cycle
ANSWER — B

After Decode, an R-type instruction needs to go to R-type execution microcode, a lw needs to go to memory address computation microcode, a beq needs branch comparison microcode, etc. There is no single "next" location that works for all instructions. The dispatch table maps the opcode to the correct µPC address for each instruction type. This corresponds to the two "branch" states in the multicycle FSM (Dispatch1 and Dispatch2) — they become table lookups in microcode.

P5 SHORT ANSWER RISC vs. CISC — the backward compatibility argument

Shankar gives the BCD (Binary Coded Decimal) instruction as an example of an instruction that persists in ISAs long after it becomes obsolete. Explain in 2–3 sentences: why is backward compatibility a con of microcode/CISC design, and why does RISC-V avoid this problem?

ANSWER

Once an instruction is added to an ISA and real programs are compiled for it, removing it breaks those programs — hardware vendors must keep supporting it forever regardless of cost. This locks in old design decisions and forces new hardware to carry dead weight (BCD instructions in x86, for example, take silicon area and add complexity even though virtually no modern software uses them). RISC-V was designed from scratch as a clean-slate ISA with no legacy code base to support, so its designers could make deliberate, principled choices without being forced to maintain obsolete features.
